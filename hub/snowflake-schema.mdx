## Databases

| Database | Description |
|---|---|
| `DBT_ANALYTICS` | SF accounts (via Fivetran), HubSpot companies, account mappings, health scores |
| `PROFOUND_GTM` | Gong call data (`GONG.FACT_GONG_CALLS`) |

<Warning>
  Always use `DBT_ANALYTICS` for SF and HubSpot data. Use `PROFOUND_GTM` **only** for Gong call data. Never write to either database — the Hub is read-only against Snowflake.
</Warning>

## `hubspot_company_id` type inconsistency

`hubspot_company_id` is stored as different types across Snowflake tables. Always cast when joining.

<Warning>
  Failing to cast `hubspot_company_id` before joining will cause silent mismatches or query failures.
</Warning>

| Table | Type |
|---|---|
| `HUBSPOT.COMPANY.ID` | NUMBER |
| `EM_ACCOUNT_MAPPING.HUBSPOT_COMPANY_ID` | NUMBER |
| `ACCOUNT_MAPPING.HUBSPOT_COMPANY_ID` | NUMBER |
| `SALESFORCE.ACCOUNT.HS_COMPANY_ID_C` | VARCHAR |
| `DAILY_ENTERPRISE_PRIORITY_SCORE.HUBSPOT_COMPANY_ID` | VARCHAR |

Always cast: `CAST(em.hubspot_company_id AS VARCHAR)`

## Three-method ID mapping (SF ↔ HubSpot)

The Hub resolves Salesforce ↔ HubSpot ID mappings using three methods in priority order:

| Priority | Method | Source | Confidence |
|---|---|---|---|
| A | SF field | `SALESFORCE.ACCOUNT.HS_COMPANY_ID_C` | 1.0 |
| B | HubSpot reverse | `HUBSPOT.COMPANY.PROPERTY_SALESFORCEACCOUNTID` | 0.9 |
| C | Domain fallback | `HUBSPOT.COMPANY.PROPERTY_DOMAIN` match | 0.7 |

The first method that returns a match wins. Results are stored in the `id_mapping` table.

<Tip>
  When debugging a missing `hubspot_company_id`, check the `id_mapping` table and look at the `match_method` and `confidence` columns to understand how (or why) the mapping was resolved.
</Tip>
